   2. Конвейер загрузок (qbittorrent + arr-ы)

### Jellyfin (CT 100) — ПРОТОКОЛ ВОССТАНОВЛЕНИЯ

Jellyfin — самый капризный. У него GPU passthrough, транскодинг, 12GB RAM.
Проблемы бывают трёх типов:

#### Тип 1: Контейнер stopped

```
ШАГ 1. Диагностика
ssh root@192.168.1.1 "pct start 100"   ← просто попробовать поднять
sleep 10
STATUS=$(ssh root@192.168.1.1 "pct status 100 | awk '{print \$2}'")

ШАГ 2. Если running — проверить сервис
ssh root@192.168.1.1 "pct exec 100 -- systemctl is-active jellyfin"
→ если active: ✅ "jellyfin был stopped, перезапущен, работает (простой 15 сек)"
→ если не active: перейти к Типу 2

ШАГ 3. Если НЕ running после pct start — смотреть причину
ssh root@192.168.1.1 "pct exec 100 -- journalctl -n 100 --no-pager" 2>/dev/null
ssh root@192.168.1.1 "dmesg | tail -30"
→ искать: OOM killer, GPU error, mount failure

ШАГ 4. OOM (Out of Memory) — самая частая причина
Признаки: в логах "Out of memory", "Killed process"
Решение:
  - Уменьшить tmpfs транскодинга если раздут:
    ssh root@192.168.1.1 "pct exec 100 -- mount -o remount,size=2G /mnt/transcode"
  - Перезапустить: ssh root@192.168.1.1 "pct stop 100 && sleep 5 && pct start 100"
  - Сообщить: "jellyfin упал по OOM, перезапущен. Рассмотри увеличение RAM до 16GB"

ШАГ 5. GPU issue (рендер не доступен)
Признаки: "/dev/dri/renderD128 not found" или ошибки VAAPI
Решение: 
  ssh root@192.168.1.1 "pct stop 100 && sleep 5 && pct start 100"
  (перезапуск контейнера переинициализирует GPU)
  Если не помогло — алерт: "GPU не доступен для jellyfin. Возможно проблема на хосте."

ШАГ 6. Если ничего не помогло — ЭСКАЛАЦИЯ
Сообщить: 
  "❌ Jellyfin не поднимается. Пробовал: pct start, перезапуск, сброс tmpfs.
   Логи: [последние 30 строк journalctl]. Нужно ручное вмешательство."
```

#### Тип 2: Контейнер running, но сервис jellyfin не active

```
ШАГ 1. Попробовать перезапустить сервис
ssh root@192.168.1.1 "pct exec 100 -- systemctl restart jellyfin"
sleep 5
ssh root@192.168.1.1 "pct exec 100 -- systemctl is-active jellyfin"
→ active? ✅ Готово. Сообщить.

ШАГ 2. Посмотреть логи сервиса
ssh root@192.168.1.1 "pct exec 100 -- journalctl -u jellyfin -n 50 --no-pager"
→ искать: port already in use, config error, permission denied, 
  database locked, ffmpeg not found

ШАГ 3. Частые причины и решения
- "port 8096 already in use": 
  ssh root@192.168.1.1 "pct exec 100 -- fuser -k 8096/tcp && systemctl restart jellyfin"
- "Permission denied" к /mnt/serial или /mnt/films:
  проверить что mount points живы:
  ssh root@192.168.1.1 "pct exec 100 -- ls /mnt/serial/ | head -3"
  если mount отвалился: pct stop 100 && pct start 100 (переподключит mount points)
- "database is locked":
  ssh root@192.168.1.1 "pct exec 100 -- rm /var/lib/jellyfin/data/jellyfin.db-shm /var/lib/jellyfin/data/jellyfin.db-wal 2>/dev/null; systemctl restart jellyfin"

ШАГ 4. Если сервис не поднимается после 3 попыток — ЭСКАЛАЦИЯ
```

#### Тип 3: Всё running, но jellyfin не отвечает на http

```
ШАГ 1. Проверить порт
ssh root@192.168.1.1 "pct exec 100 -- ss -tlnp | grep 8096"
→ нет? сервис висит: перезапустить (Тип 2)

ШАГ 2. Проверить нагрузку
ssh root@192.168.1.1 "pct exec 100 -- top -b -n1 | head -5"
→ CPU 100%? Возможно идёт транскодинг тяжёлого файла.
  НЕ перезагружать — подождать 5 минут.
  Если через 5 минут не отвечает → мягкий рестарт сервиса.

ШАГ 3. Если не отвечает и нагрузка низкая — перезапустить сервис
ssh root@192.168.1.1 "pct exec 100 -- systemctl restart jellyfin"
sleep 5
curl -s http://192.168.1.11:8096/health
→ отвечает? ✅ Сообщить: "jellyfin завис, перезапущен, работает"
```

  → если не поднялся: проверить mount points (их много, любой битый — стоп)

  → не помогло: проверить логи

  curl -s http://192.168.1.12:8090
  → не отвечает: systemctl restart qbittorrent-nox

### Sonarr / Radarr (CT 103 / CT 105)

```
Однотипные сервисы, одинаковый протокол:

1. pct start <id> (если stopped)
2. systemctl restart sonarr (или radarr) внутри CT
3. Проверить логи если не поднялся
4. Частая причина: БД повреждена при жёстком выключении
   → проверить: pct exec <id> -- ls -la /var/lib/sonarr/sonarr.db
   → если БД бита: нужна ручная очистка (сообщить владельцу)
```

### Jackett (CT 104)

```
Особенность: DHCP. После перезагрузки хоста IP может измениться.

1. pct start 104
2. Узнать новый IP:
   ssh root@192.168.1.1 "pct exec 104 -- ip -4 addr show eth0 | grep -oP 'inet \K[\d.]+'"
3. Проверить что отвечает:
   curl -s http://<новый IP>:9117
4. Сообщить: "jackett перезапущен, новый IP: x.x.x.x"
   (это важно — sonarr/radarr могут ссылаться на старый IP)
```

### Samba (на хосте PVE)

```
Не в контейнере — сервис на самом хосте:

1. Проверить: ssh root@192.168.1.1 "systemctl is-active smbd"
2. Если inactive: ssh root@192.168.1.1 "systemctl restart smbd"
3. Проверить что шары доступны:
   ssh root@192.168.1.1 "smbclient -L localhost -N 2>/dev/null | head -20"
4. Если samba упала — известить: 
   "⚠️ Samba была остановлена, перезапущена. 
    Проверь что медиатека доступна на клиентах."
   
5. Samba не поднимается — ЭСКАЛАЦИЯ немедленно.
   Без samba все mount points в CT отвалятся → каскадный сбой.
```

## Что ты НЕ делаешь

- НЕ перезагружаешь jellyfin в 18:00–01:00 MSK без алерта (там могут смотреть)
- НЕ меняешь конфиги LXC (CPU pinning, GPU passthrough)
- НЕ трогаешь /etc/samba/smb.conf
- НЕ делаешь `apt upgrade` на всех CT разом
- НЕ дублируешь команды pve-tgbot (/status, /temps он делает сам)
- НЕ спамишь. Одна проблема = одно сообщение с результатом.
- НЕ пишешь «всё хорошо» чаще раза в сутки.
